# Graph permission resources

`resources/graph_endpoint_permissions.json` and `resources/enum_command_permissions.json`
map Graph endpoints and `gts enum` subcommands to their required scopes. They are the
build-time **source** the generator compiles into the committed runtime module
`src/gts/enum/scopes.py` (which the enum guard reads) — nothing reads these JSON files at
runtime. They also back the drift check and the FOCI min-set analysis.

Both files are **generated** — never hand-edit them. Corrections go in
`scripts/permission_overrides.py`; the generator preserves the verbatim upstream values
alongside your override so divergence stays auditable.

## Do this first

Run the drift check before anything else. It is free, needs no network, and tells you
whether there is actually work to do:

```bash
uv run python .claude/skills/graph-permissions/scripts/check_drift.py
```

It AST-parses `src/gts/enum/operations.py` for `client.get*()` calls and compares them
against the generator's endpoint map, then compares the CLI's registered subcommands
against the resource keys. Exit code 1 means drift.

## Adding an endpoint or subcommand

1. Add the endpoint to `ENDPOINT_DOCS` in `scripts/build_permission_resources.py`, mapping
   it to its doc page under `api-reference/v1.0/api/`.
   **Do not guess the filename** — the naming is irregular (`GET /identity/conditionalAccess/policies`
   lives in `conditionalaccessroot-list-policies.md`, not `conditionalaccesspolicy-list.md`).
   Probe candidates instead:

   ```bash
   uv run python .claude/skills/graph-permissions/scripts/lookup_permissions.py --find conditionalaccess
   ```

2. Read the permission table without pulling the whole page into context:

   ```bash
   uv run python .claude/skills/graph-permissions/scripts/lookup_permissions.py --page group-list.md
   ```

   Doc pages run to thousands of lines of request/response samples. Use this script rather
   than WebFetch or curl — it prints ~8 lines and follows `{INCLUDE}` directives for you.

3. Add the subcommand's endpoints to `COMMAND_ENDPOINTS`, marking anything called only in
   certain branches as `"required": False` with a `conditional` string. Conditional calls
   become `optional_permissions`, so a missing scope warns instead of blocking.

4. Regenerate and verify:

   ```bash
   uv run python scripts/build_permission_resources.py
   uv run pytest -q -k "permission or endpoint or enum_all"
   ```

Bump `SOURCE_REF` in the generator to refresh against newer docs. Regeneration is
idempotent apart from the `generated` date.

## Traps in the upstream tables

These cost real time to rediscover. Check each one when adding an endpoint.

**Multiple tables per page.** The `rbacapplication-list-*` pages document one permission
table *per RBAC provider* (`### For the directory (Microsoft Entra ID) provider`,
`### For the entitlement management provider`, …). A naive parse takes the last and yields
`EntitlementManagement.Read.All` where `gts` — which only calls the directory provider —
needs `RoleManagement.Read.Directory`. Add a `DOC_SECTIONS` entry to scope the parse. The
generator refuses to guess: it errors on a multi-table page with no section.

**Two table formats.** Newer pages delegate to autogenerated
`includes/permissions/<page>-permissions.md` with three columns
(type | least | higher). Older pages carry an inline two-column table
(type | permissions from least to most privileged) marked `blockType: ignored`, where the
first entry is the least privileged. The parser handles both; the legacy ones are
hand-maintained and generally more trustworthy.

**Autogenerated least-privileged cells are sometimes wrong.** The include files say
`description: "Automatically generated file. DO NOT MODIFY"` and occasionally name an
implausible scope — `group-list-permissions.md` gives `Group-NestingSupport.ReadWrite.All`
as least privileged for `GET /groups`, where `Group.Read.All` is what you would grant.
When a value looks wrong, verify it verbatim against the raw file before overriding, then
add it to `OVERRIDES` **with a note explaining the divergence**.

**`/me` cannot be reached app-only.** `includes/me-apis-sign-in-note.md` states:
*"Application permissions aren't supported when using the `/me` endpoint."* Endpoints in
`APP_ONLY_UNSUPPORTED` get `application: null` and `app_only_supported: false`. Downstream
logic must treat these as *skip*, never as *switch to a better client* — no grant fixes them.

**`enum all` is `"partial"`, not `"unsupported"`.** `enum_all` in `cli.py` catches
`GraphError` per task and skips, so an app-only token still completes the reachable
endpoints. Marking it unsupported would wrongly tell the switching logic that no app
client can run it.

## Parsing rules

- Never summarize a permission table with an LLM. The tables are regular; prose
  summarization has already produced wrong cell attributions. Parse deterministically.
- Fail loudly on a 404 or a missing table. A silently empty entry later reads as "this
  client has sufficient permissions" when it does not — the worst failure mode here.
- `effective_least_privileged` lists **alternatives** (any one suffices).
  `required_permissions` is a list of alternative-sets: an AND of ORs, so a consumer can
  answer "does this client suffice?" with set intersection.

## FOCI minimum-set analysis

`scripts/analyze_foci_min_set.py` computes the fewest FOCI clients whose combined
delegated `.default` scopes cover `enum all`, writing `resources/foci_min_set.json`. It
reuses the required scopes from the two permission resources (the `documented` least+higher
delegated lists — "any accepted scope" coverage) and the FOCI pool from
`src/gts/constants.py`.

```bash
# one-time: fetch the gitignored scope data (see resources/entrascopes/README.md)
cd resources/entrascopes && wget https://entrascopes.com/firstpartyscopes.json && cd -
uv run python scripts/analyze_foci_min_set.py
```

Client → scope data is `resources/entrascopes/firstpartyscopes.json`, keyed
`apps.<clientId>.scopes["00000003-0000-0000-c000-000000000000"]` (Microsoft Graph). The
script never fetches it — it errors with the wget hint if absent.

Things that bite here:

- **`Directory.Read.All` is a near-universal covering scope.** It is an accepted alternative
  for every enum-all endpoint *except* conditional-access, so one client holding it covers
  almost everything. That is why the coverage model ("any accepted scope") dominates the
  result — recheck it if the answer changes.
- **conditional-access needs `Policy.Read.All`, which no FOCI client carries.** It is
  reported under `uncoverable_required_endpoints`, not silently dropped. A fully-covering
  family set does not exist; the min-set covers the coverable subset.
- **Exclusions.** The candidate pool is `constants.foci_clients()` minus any client holding
  `Directory.AccessAsUser.All` (it would trivially cover everything and mask the answer),
  reported under `excluded_clients`. Azure CLI / PowerShell need no special handling — they
  are no longer FOCI, so they never enter the pool. To change the pool, adjust that dynamic
  filter, not the algorithm.
- **Filter junk scope tokens.** Doc tables leak artifacts like `"Not available."` into the
  higher-privileged cell; `SCOPE_RE` drops anything with a space so they are not mistaken
  for grantable scopes. (The generator's `split_perms` only strips `"Not supported."` — a
  known gap.)
- The answer is **capability, not guarantee**: `.default` also depends on user/tenant
  consent, and delegated calls still need the signed-in user's directory privilege.

## Runtime consumer: the enum guard

`build_permission_resources.py` emits a **third** output beside the two JSON maps: the static
module **`src/gts/enum/scopes.py`** (`ENUM_REQUIRED_SCOPES`, `CONDITIONAL_ACCESS_SCOPES`,
`WILDCARD_SCOPE`). The `enum` guard imports it, so nothing reads `resources/` at runtime —
re-running this generator after a docs refresh is what shifts which commands the guard flags.

- **`enum` guard** — `_enum_preflight` prompts (interactive) / warns (non-interactive) when
  the active client lacks an accepted scope for a required endpoint; `enum all` hard-skips
  conditional-access unless the client has `Policy.Read.All` or `Directory.AccessAsUser.All`.
  Regenerate `enum/scopes.py` whenever the permission maps change.

Client/FOCI data (for `gts clients query`, `gts aliases`, `refresh-to`, FOCI status) is a
**separate** concern: it lives in `constants.CLIENTS`, regenerated by the `client-inventory`
skill (`scripts/build_clients.py`). It does not come from these permission resources.

`resources/entrascopes/` is third-party data from entrascopes.com (Fabian Bader,
Dirk-jan Mollema), gitignored and fetched locally. It is not generated by this process —
leave it alone.
